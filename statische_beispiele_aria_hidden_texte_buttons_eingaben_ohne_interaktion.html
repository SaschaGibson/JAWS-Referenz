Statische Beispiele: aria-hidden – Texte, Buttons & Eingabefelder

Diese Seite zeigt fest kodierte Musterfälle ohne JavaScript und ohne umschaltbare Zustände – ideal für Umgebungen ohne Interaktion.

Kurzüberblick

Grundregel: aria-hidden entfernt Knoten aus dem Accessibility-Tree, hat aber keinen Einfluss auf Sicht- oder Fokusierbarkeit. Für interaktive Elemente ist es deshalb nicht geeignet, um sie „verschwinden“ zu lassen.

1) Text-Information

Gute Praxis OK

Sichtbar: Normaler Absatztext (sichtbar & lesbar)
Deko-Span: ★ Dekoratives Symbol (mit aria-hidden="true")

Dekoratives hat keinen Informationswert → darf aus dem Accessibility-Tree entfernt werden.

<p>Normaler Absatztext</p>
<span aria-hidden="true">★ Dekoratives Symbol</span>

Schlechte Praxis NICHT OK

Sichtbar: Warnhinweis (visuell sichtbar) aber für SR unsichtbar

Wichtig: Bitte speichern, bevor Sie fortfahren.

Der Hinweis ist inhaltlich relevant, wird aber durch aria-hidden versteckt → Screenreader-Nutzer verlieren Information.

<p aria-hidden="true"><strong>Wichtig:</strong> Bitte speichern …</p>

2) Buttons

Gute Praxis OK

Button:

Deaktiviert & ohne aria-hidden. Der Button bleibt erkennbar (Role/Name), aber nicht bedienbar.

<button type="button" disabled>Aktion</button>

Schlechte Praxis NICHT OK

Button:

Mit aria-hidden="true" fehlt der Button im Accessibility-Tree, kann aber je nach Kontext weiterhin fokussierbar sein → „unsichtbarer Fokus“.

<button type="button" aria-hidden="true">Aktion</button>

Besser ausblenden OK

Entfernt: Dieser Button ist mit hidden entfernt (hier nur als Code gezeigt).
<button type="button" hidden>Aktion</button>
<!-- oder per CSS: style="display:none" -->

Bereich inaktiv OK

Container:

inert macht Inhalte un-fokussierbar & von assistiver Tech ignorierbar – nützlich z. B. hinter Modaldialogen.

<div inert>
  <button type="button">Aktion</button>
</div>

3) Eingabefelder

Gute Praxis OK


Klassisches, korrekt beschriftetes Feld. Keine versteckenden ARIA-Attribute.

<label for="name">Ihr Name</label>
<input id="name" type="text" autocomplete="name">

Schlechte Praxis NICHT OK


Die gesamte Gruppe ist aus dem Accessibility-Tree entfernt. Das Feld könnte dennoch Fokus erhalten → keine Name/Role/Value-Infos.

<div aria-hidden="true">
  <label for="email">E‑Mail</label>
  <input id="email" type="email">
</div>

Besser deaktivieren OK


Mit disabled bleibt das Feld erkennbar, ist aber nicht bedienbar.

<input id="plz" type="text" disabled>

Besser entfernen OK

Beispiel: Feld wird serverseitig gar nicht gerendert oder per hidden entfernt.

<!-- Feld nicht ausgeben (SSR) -->
<input id="bonus" type="text" hidden>

Best Practices (Kurz):

  1. aria-hidden nur für dekorative Inhalte.
  2. Interaktive Elemente niemals nur mit aria-hidden „verstecken“.
  3. Für echte Deaktivierung/Verbergung: hidden, display:none, inert, oder Element gar nicht rendern.
  4. Immer Name/Role/Value sichern: Label korrekt verdrahten, Disabled statt Verstecken, wenn Information erhalten bleiben soll.
